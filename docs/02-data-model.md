# Модель данных

## Основные сущности

### User

Пользователь приложения. Система рассчитана на работу нескольких пользователей, с изоляцией данных на уровне `user_id`.

Ключевые поля:

- `id`
- `email`
- auth/session-поля Better Auth
- `status` - `pending`, `active` или `blocked`
- `activated_at`
- `created_at`
- `updated_at`

## Доступ пользователей

Используется регистрация через email и пароль без обязательного подтверждения email.

После регистрации пользователь получает статус `pending`. Доступ к рабочим данным приложения разрешен только пользователям со статусом `active`.

Активация выполняется вручную через SQLite:

```sql
update profiles
set status = 'active', activated_at = CURRENT_TIMESTAMP
where user_id = '<user_id>';
```

Статусы:

- `pending` - пользователь зарегистрирован, но доступ еще не разрешен;
- `active` - пользователь может работать с приложением;
- `blocked` - доступ пользователя запрещен.

Роли пользователей не добавляются. Если понадобится админка для управления пользователями, можно добавить роли или отдельную модель permissions.

### Client

Клиент, к которому привязываются записи.

Ключевые поля:

- `id`
- `user_id`
- `name`
- `label` - пометка для различения клиентов с одинаковым именем
- `city`
- `phone`
- `telegram`
- `vk`
- `instagram`
- `note`
- `custom_data`
- `archived_at`
- `deleted_at`
- `created_at`
- `updated_at`
- `revision`

### Appointment

Запись/сеанс.

Ключевые поля:

- `id`
- `user_id`
- `client_id` - optional
- `starts_at`
- `title`
- `type` - `work` или `personal`
- `status` - `scheduled` или `cancelled`
- `session_amount`
- `prepayment_amount`
- `note`
- `custom_data`
- `deleted_at`
- `created_at`
- `updated_at`
- `revision`

### Attachment

Фото или файл, привязанный к записи.

Ключевые поля:

- `id`
- `user_id`
- `appointment_id`
- `storage_key`
- `original_name`
- `mime_type`
- `size_bytes`
- `created_at`
- `deleted_at`
- `revision`

## Деньги

Суммы хранятся целыми числами в минимальных единицах, например в копейках.

```text
1500 рублей -> 150000
```

Не использовать `float` для денег.

`session_amount` - полная стоимость сеанса. `prepayment_amount` - справочное поле для отображения уже внесенной предоплаты.

Сумма к доплате не хранится в базе и вычисляется в приложении:

```text
session_amount - prepayment_amount
```

Зарплата считается как сумма `session_amount` по рабочим завершенным записям. Предоплата не уменьшает зарплату.

## Правила записей

Запись может быть без клиента. Это используется для личного или служебного времени, например рисования.

`title` обязателен для всех записей, потому что запись без клиента должна понятно отображаться в списке и календаре.

Типы записей:

- `work` - рабочий сеанс, участвует в статистике зарплаты;
- `personal` - личное/служебное время, не участвует в статистике зарплаты.

В базе хранится только отмена:

- `scheduled` - запись не отменена;
- `cancelled` - запись отменена.

Отображаемый статус вычисляется без cron/job:

```text
cancelled  # если status = cancelled
completed  # если status = scheduled и starts_at <= now
scheduled  # если status = scheduled и starts_at > now
```

В зарплату попадают только записи:

```text
type = work
status != cancelled
starts_at <= now
```

## Удаление и архив

- Клиенты не удаляются физически, если у них есть записи.
- Обычное удаление клиента переводит его в архив через `archived_at`.
- Старые записи продолжают ссылаться на клиента.
- Снимок данных клиента в записи пока не храним. Если позже понадобится фиксировать историческое имя/контакты клиента на момент записи, можно добавить `client_snapshot` отдельной миграцией.
- Для очистки персональных данных можно позже добавить отдельное действие: очистить контакты и заметки, но оставить обезличенную историю.

## Расширяемость

Стабильные поля должны быть колонками. Нестандартные и будущие поля хранятся в `custom_data`.

Примеры стабильных полей:

- дата и время записи;
- сумма сеанса;
- предоплата;
- клиент;
- статус удаления/архива;
- timestamps;
- revision.

Примеры расширяемых полей:

- источник клиента;
- тип услуги;
- дополнительные заметки;
- любые пользовательские признаки.
